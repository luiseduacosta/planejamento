# PlanejamentosController — Null-Id Guards & Regression Tests

## Source Request
Five methods in `PlanejamentosController` lacked null/non-existent ID checks, causing undefined-index warnings, silent false-saves, and unpredictable `read()` behavior. The fixes are already applied in code; this plan documents them, adds regression tests, and hardens the remaining surface.

## Summary
Add formal CakePHP 2.x controller test cases for `view()`, `delete()`, `edit()`, `relaciona()`, and `excluir()` to lock in the null-ID guards already applied. No production code changes needed — the fixes are in place. Deliver 1 test file with ~10 test methods covering the five patched actions plus the two intentionally non-blocking conflict-warning paths (`add()` and `edit()`).

## Requirements

### R-1: `view($id)` must reject null/missing IDs {implemented}
**Status:** Fixed in code.  
When `$id` is `null`, empty, or does not exist in the database, `view()` must flash an error and redirect to `Planejamentos/listar` instead of passing an empty result set to the view.

Acceptance:
- [x] `view(null)` → Flash error + redirect
- [x] `view(99999)` (non-existent) → Flash error + redirect
- [x] `view(valid_id)` → normal render

### R-2: `delete($id)` must reject null/missing IDs {implemented}
**Status:** Fixed in code.  
When `$id` is `null` or the record does not exist, `delete()` must flash an error and redirect. Must not execute `$this->Planejamento->delete(null)` (which returns `false` but previously displayed a success message anyway).

Acceptance:
- [x] `delete(null)` → Flash error + redirect
- [x] `delete(99999)` (non-existent) → Flash error + redirect
- [x] `delete(valid_id)` → delete + Flash success

### R-3: `edit($id)` must reject null/missing IDs {implemented}
**Status:** Fixed in code.  
When `$id` is `null` or the record does not exist, `edit()` must flash an error and redirect. Must not call `$this->Planejamento->read()` with ID `null`, which caused unpredictable behavior.

Acceptance:
- [x] `edit(null)` → Flash error + redirect
- [x] `edit(99999)` (non-existent) → Flash error + redirect
- [x] `edit(valid_id)` → normal form render

### R-4: `relaciona($id)` must reject null/missing IDs {implemented}
**Status:** Fixed in code.  
Same pattern as `edit()` — `$this->Planejamento->read()` with a null ID causes unpredictable behavior. Must validate `$id` before calling `read()`.

Acceptance:
- [x] `relaciona(null)` → Flash error + redirect
- [x] `relaciona(99999)` (non-existent) → Flash error + redirect
- [x] `relaciona(valid_id)` → normal form render

### R-5: `excluir($id)` must reject null IDs and empty result sets {implemented}
**Status:** Fixed in code.  
When `$id` is `null`, the `deleteAll`-equivalent loop must not run with a null FK condition. When no records are found, must flash an error instead of silently succeeding.

Acceptance:
- [x] `excluir(null)` → Flash error + redirect
- [x] `excluir(valid_id_with_no_planejamentos)` → Flash "Nenhum registro encontrado" + redirect
- [x] `excluir(valid_id_with_planejamentos)` → deletes all matching records + Flash success

## Non-Bugs (intentional behavior — verified)

### `add()` and `edit()` conflict detection
The `_buildGrade()` method stores multiple records per cell using `$mapa[$per][$hor][$dia][]`, deliberately allowing multiple entries in the same slot. The `$existe` check in `add()`/`edit()` is informational only — the save always proceeds. The flash message warns the user but does not block. This is by design.

**Verification:** Tests must confirm that `add()` and `edit()` **do** save even when `$existe` is truthy, and that the correct flash message (warning for edit, error for add — the existing wording is preserved) is set.

### Tigers (real problems — act on them)

| # | Tiger | Class | Mitigation | Owner | Decision Date |
|---|-------|-------|------------|-------|---------------|

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| CakePHP 2.x `ControllerTestCase` mocking API is fragile with deep model associations | Medium | Use `$this->generate()` with `'models' => []` to mock all models, or use real fixtures with a test DB |
| Fixtures may need `$import` to match live schema changes | Low | Use `$import = ['table' => 'planejamentos', 'connection' => 'test']` so fixture inherits schema automatically |
| Tests may be slow if they hit a real DB | Low | Point test DB at SQLite in-memory or a dedicated test database configured in `Config/database.php` |
| `Auth` component may interfere with test requests | Medium | Use `$this->testAction()` with `'return' => 'vars'` and mock `Auth` in `generate()` |
